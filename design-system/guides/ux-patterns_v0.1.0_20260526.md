## 1. 색 사용 원칙

### Primary
- 한 화면의 *주요 액션 1개*에만 사용 (제출·저장·시작).
- 여러 Primary 버튼이 동시에 있으면 시각 위계가 무너져 가독성 저하.
- 보조 액션은 `default` 또는 `dashed` 버튼으로.

### Danger
- *되돌릴 수 없는* 액션 (삭제·계정 해지·결제 취소)에만 사용.
- 모달의 확인 버튼에 danger 적용 시, 모달 본문도 위험성을 명시.

### Semantic (Success / Warning / Info)
- 결과 알림·배지·태그에 한정. 인터랙티브 컨트롤(버튼)에는 직접 사용 자제.
- 색만으로 의미를 전달하지 말 것 (P-06): 아이콘·텍스트 병행.

### Surface 위계
- `surface.base` — 페이지 캔버스 (가장 낮은 레이어)
- `surface.elevated` — 카드·모달·드로어 (한 단계 위)
- `surface.muted` — hover·disabled·subtle 배경
- 라이트 모드에선 base/elevated가 거의 같은 흰색이라도 그림자·테두리로 구분.
- 다크 모드에선 명도 차이로 구분 (그림자가 잘 안 보임).

## 2. 폼 레이아웃

### 권장 구조
1. **그룹화** — 관련 필드를 `<fieldset>` 또는 `Section` 단위로 묶기.
2. **단일 컬럼** — 좁은 화면 대응. 짧은 필드만 가로 2열 허용.
3. **명시적 라벨** — placeholder 단독 금지 (KWCAG 4.1).
4. **인라인 오류** — 필드 직후 + `aria-describedby` 연결.
5. **명시적 제출** — Enter 키 + Submit 버튼 모두 작동.

### 안티 패턴
- 포커스 이동만으로 자동 제출 (WCAG 3.2.2).
- 라벨 없이 placeholder만 (스크린리더 의미 손실).
- 한 화면에 10개 초과 필드 (분할·단계화 검토).

## 3. Empty / Loading / Error 상태

리스트·테이블·검색은 *최소 4가지 상태*를 모두 고려:

| 상태 | 컴포넌트 | 메시지 톤 |
|---|---|---|
| 로딩 중 | `Skeleton` (행 단위) | 콘텐츠 윤곽 |
| 데이터 없음 (첫 사용) | `Empty` + CTA | 다음 행동 유도 |
| 데이터 없음 (필터 결과) | `Empty` + "필터 재설정" | 조건 변경 안내 |
| 오류 | `Result` (danger) | 원인 + 재시도 |

각 상태는 동일한 영역에 배치해 *레이아웃 시프트(CLS)* 최소화.

## 4. 알림 위계

| 컴포넌트 | 지속 시간 | 차단성 | 사용 케이스 |
|---|---|---|---|
| `Toast` | 5초 자동 사라짐 | 비차단 | 결과 알림 ("저장됨") |
| `Alert` | 영속 (사용자 닫기) | 비차단 | 페이지 내 검증 오류 요약 |
| `Modal` | 사용자 결정까지 | 차단 | 되돌릴 수 없는 액션 확인 |
| `Drawer` | 사용자 닫기 | 부분 차단 | 필터·상세 보기 |
| `Banner` (v0.3) | 영속 | 비차단 | 전역 공지·점검 안내 |

**원칙**: 차단성이 높을수록 사용자 명시적 결정이 필수. 일시 알림과 차단 메시지를 혼동하지 말 것.

## 5. 다크 모드 사용

- 토큰 파일은 `[data-theme="dark"]` 속성 + `@media (prefers-color-scheme: dark)` 둘 다 매핑됨.
- 사용자가 명시적으로 `[data-theme="light"]`를 설정하면 OS 설정과 무관하게 라이트 유지.
- 다크 모드에서:
  - `surface.base`는 거의 검정, `surface.elevated`는 한 단계 밝아져 카드가 떠 보임.
  - `primary`·`error` 액션 배경은 *밝게*, 라벨은 *어둡게* (GitHub Dark 패턴).
  - 그림자 효과는 거의 안 보이므로 elevation을 *명도 차이*로 표현.

### 다크 모드 토글 예시
```html
<!-- 사용자가 토글 버튼으로 전환 -->
<html data-theme="dark"> ... </html>

<!-- OS 설정 따라가기 (기본) -->
<html> ... </html>
```

## 6. 밀도 (Density)

| 단계 | 권장 사용처 | `control.height.md` |
|---|---|---|
| **촘촘 (compact)** | 데이터 헤비 어드민·로그·모니터링 | 28px |
| **기본 (default)** | 대부분의 화면 | 32px |
| **여유 (comfortable)** | LMS 학습·B2C 모바일·접근성 우선 | 40px |

**현재 프로젝트 적용값**: `default` (control.height.md = 32px)

터치 타겟 최소 24×24px(WCAG 2.5.8)은 모든 단계에서 자동 보장.

## 7. 접근성 핵심 5가지

1. **대비** — 본문 ≥ 4.5:1, 대형 텍스트·UI 컨트롤 ≥ 3:1. 자동 보정 도구가 미달 시 제안.
2. **키보드** — 모든 인터랙티브 요소가 Tab·Enter·Space·Esc로 작동.
3. **포커스 가시** — 포커스 링 2px 이상 + 배경 대비 3:1 이상.
4. **의미 전달** — 색만으로 의미 전달 금지. 아이콘·텍스트·패턴 병행.
5. **한국어 타이포** — Pretendard 폰트, 자간 -0.05em~0em, 행간 ≥ 1.5.

자세한 점검 결과는 `guides/accessibility-report_*.md` 참조.

## 8. 컴포넌트 선택 결정 트리

**"옵션 중 하나 선택"이 필요할 때:**
- 옵션 2~7개 + 항상 보임 → `Radio` 또는 `SegmentedControl`
- 옵션 8개 이상 → `Select`
- 콘텐츠 영역 자체를 전환 → `Tabs`
- 즉시 토글 ON/OFF → `Switch`

**"부유 콘텐츠"가 필요할 때:**
- 텍스트 보조 라벨 → `Tooltip` (호버)
- 옵션 리스트 → `Menu` (클릭)
- 임의 폼·콘텐츠 → `Popover` (클릭)
- 사용자 결정 차단 필요 → `Modal`
- 측면 패널 → `Drawer`

**"리스트 표시"가 필요할 때:**
- 동일 형태 행 + 모바일 친화 → `List`
- 다열 비교·정렬·필터 → `Table` (v0.3+)
- 계층 구조 → `Tree` (v0.3+)

## 9. UX 원칙 요약 (Why behind the design)

본 가이드의 패턴은 다음 10가지 UX 원칙에 근거합니다. 인수자가 *결정의 이유*를
이해할 수 있도록 한 줄 요약과 적용 매핑만 정리하며, 깊이 있는 설명은 외부 출처를 참조하세요.

### 핵심 10가지

| # | 원칙 | 한 줄 요약 |
|---|---|---|
| 1 | 힉의 법칙 (Hick's Law) | 선택지가 많을수록 결정 시간이 늘어난다 |
| 2 | 피츠의 법칙 (Fitts's Law) | 멀고 작을수록 누르기 어렵다 |
| 3 | 제콥의 법칙 (Jakob's Law) | 사용자는 익숙한 패턴을 기대한다 |
| 4 | 밀러의 법칙 (Miller's Law) | 단기기억은 7±2 청크만 |
| 5 | 심미적 사용성 효과 (Aesthetic-Usability) | 예쁘면 잘 작동하는 것처럼 느낀다 |
| 6 | 도허티 임계값 (Doherty Threshold) | 400ms 안에 반응해야 몰입이 유지된다 |
| 7 | 피크-엔드 법칙 (Peak-End Rule) | 절정과 끝만 기억된다 |
| 8 | 인지 부하 (Cognitive Load) | 한 번에 한 가지 결정만 |
| 9 | 게슈탈트 원리 (Gestalt) | 근접·유사·폐쇄·연속·공통운명 |
| 10 | F·Z 읽기 패턴 (Reading Patterns) | 시선의 기본 동선 |

### 결정 상황 ↔ 원칙 매핑

| 결정 상황 | 관련 원칙 | 본 가이드 참조 |
|---|---|---|
| 메뉴·옵션 개수 | 밀러·힉 | §8 컴포넌트 선택 결정 트리 |
| 버튼 크기·위치 | 피츠 | §6 밀도 |
| 로딩·반응 속도 | 도허티 | §3 Empty/Loading/Error |
| 익숙한 배치 (로고·검색·취소) | 제콥 | §2 폼 레이아웃 |
| 컴포넌트 그룹화 | 게슈탈트 | §2 폼 레이아웃 |
| 시각 강조 (Primary 1개) | 심미적 사용성 | §1 색 사용 원칙 |
| 빈 상태·성공 화면 | 피크-엔드 | §3 Empty/Loading/Error |
